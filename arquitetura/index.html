    Arquitetura de Dados SkyData

    Nossa arquitetura segue o padrão Medallion Architecture (Bronze, Silver, Gold), apoiada por uma camada de controle de metadados para garantir robustez e rastreabilidade.

    1. Estratégia de Controle (Meta Layer)

    Para orquestrar a ingestão incremental sem duplicidade, criamos um esquema dedicado de auditoria: aviacao_meta.

    • Tabela de Controle (silver_bronze_watermark): Armazena o "estado" de cada tabela processada.
      • ultima_data_ref: A data do registro mais recente processado.
      • ultima_execucao_ts: Quando o pipeline rodou pela última vez.
    • Funcionamento: Antes de processar, o pipeline consulta essa tabela para saber "de onde parou". Após o sucesso, ele atualiza essa marca d'água.

    2. Pipeline de Ingestão (Landing & Bronze)

    • Fonte: PostgreSQL (simulado com Faker).
    • Landing: Ingestão via JDBC. O script lê apenas registros onde AUD_DH_ALTERACAO > último watermark salvo.
    • Bronze: Os dados são salvos em formato Delta Lake com operação MERGE (Upsert), garantindo que a Bronze seja um espelho fiel e histórico do banco transacional.

    3. Pipeline de Refinamento (Silver Layer)

    Esta camada é o coração da inteligência temporal do projeto. O script foi desenvolvido de forma Metadata Driven:

    • Configuração Centralizada: Um dicionário (TABLE_CONFIGS) mapeia cada tabela às suas chaves de negócio (Business Keys), permitindo que um único código processe múltiplas tabelas.
    • Algoritmo SCD Tipo 2 (Passo a Passo):
      1. Leitura Incremental: O sistema lê da Bronze apenas o que mudou desde a última execução (bronze_load_ts > watermark).
      2. Hashing: Gera um attr_hash (SHA-256) das colunas de negócio.
      3. Comparação: Cruza os dados novos com a tabela Silver atual (filtrando is_current = true).
      4. Merge (SCD2):
        • Se o hash mudou: "Fecha" o registro antigo (atualiza vigencia_fim e is_current = false) e insere o novo.
        • Se é registro novo: Apenas insere.
      5. Commit: Atualiza a tabela de controle no schema META.

    4. Camada de Negócio (Gold Layer)

    Camada de consumo modelada em Star Schema (Esquema Estrela) seguindo metodologia Kimball.

    4.1 Dimensões

    • dim_tempo: Calendário canônico.
    • dim_companhia, dim_aeroporto, dim_aeronave: Dimensões com histórico (SCD2).
    • dim_cliente: Perfil dos passageiros.

    4.2 Fatos

    • fato_voo: Métricas de atraso, passageiros e status.
    • fato_pagamento: Receita detalhada.
    • fato_manutencao: Custos e tempo de parada.
    • fato_bagagem: Peso e volume.

    4.3 Regras de Negócio

    • KPIs: Calculados diretamente nas tabelas fato ou views agregadas.
    • Tratamento de Nulos: Valores nulos em dimensões são mapeados para registros "Não Informado" (-1) para manter integridade referencial.

    5. Tecnologias & Visualização

    • Databricks Volumes: Gestão de arquivos brutos.
    • Delta Lake: Armazenamento ACID.
    • Apache Spark (PySpark): Motor de processamento.
    • Dashboard: Painéis conectados diretamente à camada Gold para visualização de KPIs (Pontualidade, Receita, Ocupação).
